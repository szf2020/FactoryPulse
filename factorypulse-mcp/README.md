# FactoryPulse MCP Server

Exposes FactoryPulse's read-only OEE/downtime analytics as
[MCP](https://modelcontextprotocol.io) tools, so any MCP-compatible LLM client
(Claude Desktop, the MCP Inspector, custom agents) can query the shop floor in
natural language — "What's the OEE of DB-01 this week?", "Which machine lost
the most time to stoppages today?".

## Tools

| Tool | Description |
|---|---|
| `list_machines()` | Lists every machine: device id, name, type, ideal cycle time, current OEE snapshot |
| `get_oee(machine_id, period="24h")` | Availability / Performance / Quality / Global OEE for one machine over a period |
| `get_downtime(machine_id, period="24h")` | Stoppages for one machine — start/end/duration, including ongoing ones |
| `top_downtime(period="24h", limit=5)` | Ranks machines by total downtime within a period |

`period` accepts `"30m"`, `"24h"`, `"7d"`, `"30d"`, etc. — the same format the underlying FactoryPulse API understands.

## How it talks to FactoryPulse

`factorypulse_mcp/client.py` is a thin async `httpx` wrapper around the
read-only endpoints added to the Django API
(`/machines/`, `/machines/<id>/oee/`, `/machines/<id>/downtime/`,
`/downtime/top/`). Authentication mirrors the project's existing scheme —
DRF + SimpleJWT — sending `Authorization: Bearer <token>`, exactly like the
React frontend does. The token is the long-lived, read-only service JWT
generated by:

```bash
docker compose exec backend python manage.py create_service_token
```

## Configuration

Copy `.env.example` to `.env` and fill it in:

```bash
cp .env.example .env
```

| Variable | Description |
|---|---|
| `FACTORYPULSE_API` | Base URL of the FactoryPulse REST API, e.g. `http://localhost:8000/api` |
| `FACTORYPULSE_TOKEN` | Read-only JWT from `create_service_token` (never commit this) |
| `MCP_TRANSPORT` | `stdio` (default — Claude Desktop / `mcp dev`) or `sse` / `streamable-http` (containerized/HTTP) |
| `MCP_HOST` / `MCP_PORT` | Bind address for HTTP-based transports (default `0.0.0.0:8000`) |

## Setup

```bash
cd factorypulse-mcp
python -m venv venv
./venv/Scripts/activate      # Windows — use `source venv/bin/activate` on Linux/macOS
pip install -r requirements.txt
cp .env.example .env         # then edit FACTORYPULSE_TOKEN
```

## Inspecting it interactively (MCP Inspector)

With the FactoryPulse stack running (`docker compose up` from the repo root),
launch the [MCP Inspector](https://github.com/modelcontextprotocol/inspector):

```bash
mcp dev factorypulse_mcp/server.py
```

This opens a browser UI where you can call each tool, inspect the JSON Schema
generated from its type hints/docstring, and see the live JSON responses
coming from the FactoryPulse API — the fastest way to verify the server end
to end before wiring it into an LLM client.

To run it as a plain stdio server (what Claude Desktop launches under the hood):

```bash
python -m factorypulse_mcp.server
```

## Plugging into Claude Desktop

Open **Settings → Developer → Edit Config** and add an entry under `mcpServers`:

```json
{
  "mcpServers": {
    "factorypulse": {
      "command": "/absolute/path/to/factorypulse-mcp/venv/Scripts/python.exe",
      "args": ["-m", "factorypulse_mcp.server"],
      "env": {
        "FACTORYPULSE_API": "http://localhost:8000/api",
        "FACTORYPULSE_TOKEN": "<paste the read-only service token here>"
      }
    }
  }
}
```

(On Linux/macOS point `command` at `venv/bin/python` instead.) Restart Claude
Desktop and ask things like *"What's the OEE of DB-01 in the last 7 days?"* or
*"Which machine had the most downtime today?"* — Claude will call the tools
above and reason over the JSON they return.

## Running with Docker

```bash
docker build -t factorypulse-mcp .
docker run --rm -p 8000:8000 \
  -e FACTORYPULSE_API=http://host.docker.internal:8000/api \
  -e FACTORYPULSE_TOKEN=<token> \
  factorypulse-mcp
```

The image defaults to the `streamable-http` transport (set in the Dockerfile)
since a container can't be attached to a parent process's stdio the way
Claude Desktop attaches to a local process — `stdio` remains the default for
local/desktop use.

## Tests

```bash
pytest
```

Covers configuration parsing, the HTTP client (mocked transport — no live API
needed) and that every tool delegates to the client with the right arguments
and a descriptive docstring (the schema the LLM sees).
